# 뉴스 이슈 감지 구조 재설계 메모 2026-05-09

이 문서는 현재 뉴스 자동화 watcher 구조를 어떻게 바꿀지 논의한 내용을 정리한 메모다.

## 현재 구조 판단

현재 자동화는 아래 흐름이다.

```text
30분마다 watch-cycle 실행
-> 실행 시점 기준 최근 24시간 기사 수집/조회
-> 후보 주제 선정
-> Discord로 제작 여부 확인
-> 제작 승인 시 editor build 및 카드뉴스 제작 흐름 진행
```

현재 구조는 "이슈 발생 감지"라기보다 "주기적으로 최근 기사들을 훑고 카드뉴스 후보를 추천"하는 방식에 가깝다.

## 현재 후보 선정 기준

현재 수집 대상 언론사는 5개다.

```text
스포츠조선
스포츠한국
일간스포츠
스타뉴스
뉴스1 스포츠
```

후보 선정은 크게 아래 기준을 쓴다.

```text
최근성
기사 수
출처 다양성
조회수/engagement 값
제목/요약 키워드
같은 팀/비슷한 키워드 기반 topic clustering
이미 제작한 topic 제외
```

후보 생성 이후 automation ranker가 다시 카드뉴스 적합도를 보정한다.

가산되는 대표 신호:

```text
부상/말소/수술/이탈
논란/징계/사과/발언
끝내기/역전/연승/스윕
기록/신기록/홈런/세이브
복귀/영입/방출/외국인/엔트리
KIA/한화/LG/롯데 등 account fit이 높은 구단
```

감점되는 대표 신호:

```text
단일 기사만 있는 주제
순위표/경기결과 종합/선발 예고/라운드업
퓨처스/2군 기사
이벤트/상품/행사/중계성 기사
```

## Gemini 사용 문제

현재 `.env`에 `GEMINI_API_KEY`가 있으면 후보 선정 단계에서도 Gemini가 사용된다.

즉, 30분마다 실행되는 watcher가 기사 후보 선정 목적으로 Gemini API를 호출할 수 있다.

문제:

```text
하루 최대 약 48회 후보 선정 호출 가능
정작 카드뉴스 제작 단계에서 써야 할 quota를 후보 선정에서 소모
후보 선정은 반복 작업이라 API 비용/사용량 대비 효율이 낮음
```

결론:

```text
watcher 단계에서 모든 후보 선정에 Gemini를 쓰는 구조는 피한다.
다만 로컬 rule을 통과한 소수 fresh 후보는 Gemini로 2차 적합성 검증을 한다.
Gemini는 "대량 후보 선정기"가 아니라 "Discord 알림 전 최종 게이트"로 사용한다.
```

## Gemma 로컬 AI 검토

서버용 맥북 사양:

```text
MacBookPro14,1
Dual-Core Intel Core i5 2.3 GHz
2 cores / hyper-threading
Memory 8 GB
```

이 사양에서는 Gemma 같은 로컬 LLM을 watcher에 상시 붙이는 것은 비추천이다.

이유:

```text
CPU 추론이라 느릴 가능성이 큼
Chrome headless, Discord worker, cloudflared, Python pipeline과 자원 경쟁
8GB 메모리에서 상시 LLM 서버 운영 부담
한국어 기사 판단 품질이 Gemini보다 낮을 수 있음
운영 복잡도가 늘어남
```

쓸 수 있다면 2차 보조 판단기 정도가 적절하다.

```text
로컬 rule로 후보를 5~10개까지 압축
-> 작은 quantized 모델이 짧은 요약/분류만 수행
```

하지만 1차 개선은 Gemma 도입이 아니라 watcher를 AI 없이 이슈 감지형으로 바꾸는 것이다.

## 새 방향

새 방향은 5분 단위 fresh-window 감지다.

예:

```text
11:20에 코드 실행
-> 11:15~11:20 사이 발행된 기사만 새 기사로 수집
-> 새 기사가 없으면 종료
-> 새 기사가 있으면 이슈성 평가
-> 로컬 rule 기준 이상이면 Gemini 2차 판단
-> Gemini가 적합하다고 판단한 경우에만 Discord 알림
-> 사용자가 제작 승인하면 카드뉴스 제작 단계에서 다시 Gemini 사용
```

객관적으로 현재 30분/24시간 후보 추천 방식보다 이 방향이 더 낫다.

이유:

```text
알림의 의미가 "방금 생긴 이슈"로 명확해짐
오래된 이슈가 반복 후보로 남는 문제 감소
Gemini 호출 대상을 소수 후보로 줄여 quota 절약
이슈 발생 반응성 개선
중복 제거가 쉬워짐
```

단점:

```text
발행 시간이 늦게 찍히거나 수집기가 놓치면 빠질 수 있음
직전 5분 기사만 보면 이슈 규모 판단이 약할 수 있음
단발성 기사에 과민반응할 수 있음
```

따라서 수집 window와 판단 context를 분리한다.

```text
수집 대상: 직전 5분 또는 5~10분 발행 기사
평가 맥락: 최근 24시간 관련 기사
판단 feature: 최근 30분/1시간/3시간/6시간/24시간 확산 흐름
중복 기준: 최근 24~72시간 이미 보낸 이슈
```

## 권장 최종 구조

```text
5분마다 watcher 실행
-> 직전 5~10분 발행 기사 수집
-> 새 기사 URL/ID 중복 제거
-> 새 기사에서 팀/선수/키워드/entity 추출
-> 최근 24시간 DB에서 관련 기사 조회
-> 시간대별 확산 흐름과 로컬 rule 기반 issue_score 계산
-> 기준 이상 후보만 Gemini 2차 판단
-> Gemini approve 후보만 Discord 알림
-> 제작 버튼 클릭 시 Gemini 기반 제작 pipeline 실행
```

## 로컬 1차 이슈 평가 기준 초안

이슈 점수는 Discord 알림 여부를 최종 결정하지 않는다.
Gemini 2차 판단에 넣을 후보를 압축하기 위한 1차 필터로 사용한다.

```text
issue_score =
  새 기사 수 점수
+ 매체 다양성 점수
+ 최근성 점수
+ 강한 키워드 점수
+ 인기 구단/선수 보정
+ 최근 30분/1시간/3시간/6시간/24시간 관련 기사 증가량
- 단순 경기결과/프리뷰/라운드업 감점
- 이미 보낸 이슈/유사 URL/유사 topic 감점
```

강한 키워드 후보:

```text
부상
말소
수술
복귀
트레이드
방출
영입
외국인
징계
논란
사과
끝내기
역전
신기록
연승
연패
```

알림 예시:

```text
[강한 이슈] LG 오스틴 말소
점수: 82
근거: 최근 1시간 4건 / 3개 매체 / 말소 키워드 / LG 인기 구단
리스크: 단일 발표성 기사 아님
```

## 언론사 확대 방향

5분 단위 이슈 감지 구조에서는 언론사 수를 늘리는 것이 도움이 된다.

목적은 "더 많이 보내기"가 아니라 "더 정확히 걸러내기"다.

언론사 확대 이유:

```text
같은 선수/팀/키워드가 여러 매체에서 동시에 뜨는지 확인 가능
source diversity를 이슈 신뢰도 신호로 사용 가능
단일 기사와 실제 확산 이슈를 구분하기 쉬움
최근 24시간 맥락과 시간대별 확산 흐름을 같이 볼 수 있음
```

운영 원칙:

```text
수집 매체는 늘린다
Discord 알림 기준은 더 엄격하게 한다
새 기사 window는 작게 둔다
평가 context는 넓게 둔다
매체별 신뢰도/성격을 점수에 반영한다
```

언론사 분류 예시:

```text
1차 핵심 매체:
  발행 속도가 빠르고 KBO 현장/단독 기사 비중이 높은 매체

2차 확인 매체:
  같은 이슈가 확산되는지 확인하는 매체

저품질/라운드업 많은 매체:
  수집은 하되 점수는 낮게 보거나 특정 article_kind는 감점
```

## 핵심 결론

```text
기존 구조:
30분마다 최근 24시간 기사 중 카드뉴스 후보 추천

새 구조:
5분마다 방금 나온 기사에서 이슈 후보를 감지하고,
로컬 rule과 Gemini 2차 판단을 거쳐 Discord 알림 여부 결정
```

새 구조가 현재 목표에 더 맞다.

Gemini는 watcher 전체 후보 선정에 쓰지 않는다.
로컬 rule로 압축된 fresh 후보의 최종 적합성 검증과 제작 승인 이후 품질 개선에 쓴다.
판단 기준은 고정값이 아니라 승인/거절/제작 성과/오탐/누락 데이터에 따라 계속 보정한다.
